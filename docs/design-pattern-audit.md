# Design & Programming Pattern Audit Report

## Overview

The codebase shows clear signs of rapid iteration: the core abstractions (`Agent`, `LLMProvider`, `ToolRegistrationProtocol`) exist on paper but are bypassed by concrete imports in `FrameworkClient`, `OpenAIAgent`, and especially `TUIApp`. The event system is duplicated, error handling between subprocess tools and in-process tools is bifurcated, and several critical resource-management and security issues (`eval`, unclosed streams, zombie subprocesses) need immediate attention.

**Recommended priority:**
1. **Critical first** — Fix resource leaks, security issues, and abstraction leaks to make the framework safe and genuinely pluggable.
2. **Major next** — Deduplicate the event system, centralize configuration, and split the `TUIApp` god object to improve maintainability.
3. **Minor last** — Clean up style inconsistencies and micro-optimizations.

## Critical Issues (Correctness, Security, Extensibility)

| # | Problem | Location | Refactor Suggestion | Why |
|---|---------|----------|---------------------|-----|
| 1 | **TUI directly peeks into OpenAI-specific chunk shapes** — `event.chunk.choices[0].delta`, `delta.reasoning_content`, `delta.tool_calls`, etc. are hard-coded in the presentation layer, making the TUI unusable with any other LLM provider. | `src/minimal_harness/client/built_in/app.py` (lines 489–515) | Normalize provider-specific deltas into a provider-agnostic `LLMChunk` dataclass inside the LLM layer. `LLMChunkEvent` should carry only generic fields (`content`, `reasoning`, `tool_call_deltas`). The TUI should consume these generic fields. | Closes the abstraction leak and makes the TUI truly pluggable, which is the stated purpose of the `LLMProvider` protocol. |
| 2 | **`OpenAIAgent` is locked to `OpenAILLMProvider` despite an `LLMProvider` protocol** | `src/minimal_harness/agent/openai.py` (line 34) | Change the type hint to `llm_provider: LLMProvider`. | Allows swapping LLM backends without touching agent or client code. |
| 3 | **`TUIApp` is a God Object** — It mixes UI, business logic, and infrastructure (theming, slash commands, agent/LLM/memory stack rebuilding, SVG export, memory dump, tool selection, config persistence). | `src/minimal_harness/client/built_in/app.py` (lines 68–647) | Introduce an `AppContext` or `ClientFactory` that owns configuration, registry, and agent lifecycle, and inject it into `TUIApp`. The TUI should only call methods on the context. | Separating concerns makes the TUI unit-testable, reduces the class size by ~60%, and prevents UI changes from breaking business logic. |

## Additional Findings from built_in/ Review

The files in `src/minimal_harness/client/built_in/` were reviewed in full (`app.py`, `buffer.py`, `config.py`, `context.py`, `modals.py`, `tui.py`, `widgets.py`, `__init__.py`). Issues are categorized below. References to items from the original report (e.g. "[#1]") indicate direct elaboration of a pre-existing finding.

### Critical Issues (Correctness, Security, Extensibility)

| # | Problem | Location | Refactor Suggestion | Why | Status |
|---|---------|----------|---------------------|-----|--------|
| 7 | **Three unclosed `Console`/`StringIO` pairs — resource leaks** — `_render_markdown` (line 280–284) creates a `StringIO` and `Console` per markdown render and never closes either. `action_share` (line 534–546) does the same inside the `done` callback. `StreamBuffer.render` (buffer.py line 31–34) does this on every streaming tick. | `src/minimal_harness/client/built_in/app.py` (lines 280–284, 534–546); `src/minimal_harness/client/built_in/buffer.py` (lines 31–34) | Use context managers (`with StringIO() as buf: with Console(file=buf, ...) as console:`) or explicitly call `buf.close()` and `console.finish()` after use. For `StreamBuffer.render`, pass a shared `Console` instance or pool one. | Long-running TUI sessions that display many markdown messages or export SVGs will gradually leak file descriptors and memory. | **Resolved** — all three sites now use `with Console(...)` context managers. |
| 8 | **`AppContext` hardcodes the entire OpenAI stack** — `rebuild()` directly instantiates `AsyncOpenAI`, `OpenAILLMProvider`, and `OpenAIAgent` with no abstraction layer. This locks the entire client to OpenAI and prevents any other LLM from being used regardless of the `LLMProvider` protocol. | `src/minimal_harness/client/built_in/context.py` (lines 8–9, 54–75) | Replace the concrete imports with the `LLMProvider` and `Agent` protocol types. Accept an optional `llm_provider_factory: Callable[[dict, ToolRegistry], LLMProvider]` and `agent_factory: Callable[[LLMProvider, list[Tool], ConversationMemory], Agent]` constructor parameter on `AppContext`. | Extensibility is completely blocked; adding Anthropic, local models, or test providers requires modifying `AppContext` instead of being a drop-in swap. | Open |
| 9 | **`_on_chunk` hard-codes OpenAI delta structure** (elaboration of [#1]) — The method at lines 479–505 reaches into `event.chunk.choices[0].delta` and directly reads `reasoning_content`, `content`/`text`, and `tool_calls` by attribute name. Any non-OpenAI provider will produce a silent no-op (no content displayed) or `AttributeError`. The existing report cited lines 489–515; the actual implementation spans lines 479–505 with two additional `getattr` guards at 485–487. | `src/minimal_harness/client/built_in/app.py` (lines 479–505) | Normalize the delta in the LLM provider layer before it reaches the event bus. `LLMChunkEvent.chunk` should be a plain dataclass (`content: str`, `reasoning: str`, `tool_call_deltas: list[ToolCallDelta]`) rather than a provider-specific response object. | Same rationale as [#1]: the TUI cannot be provider-agnostic as long as raw OpenAI shapes drive rendering. | Open |
| 10 | **`action_share` has incomplete exception safety** — The `done` callback (line 527–552) creates a `Console` and `StringIO` inside a `try` block. If any exception fires before `console.export_svg()` (line 546), the `console` and `buf` are not closed, and control returns without `self.say` being called, silently failing the export. | `src/minimal_harness/client/built_in/app.py` (lines 527–552) | Move resource creation outside the try block and add a `finally` block that calls `buf.close()` and `console.finish()` unconditionally. Call `self.say` for both success and failure in the finally or after the try. | Users may attempt to export and receive no feedback when it fails, with resources leaked in the background. | **Resolved** — resources created outside try block; `with console:` ensures cleanup; `self.say` called in both success and except paths. |

### Major Issues (Maintainability, Architecture, Duplication)

| # | Problem | Location | Refactor Suggestion | Why |
|---|---------|----------|---------------------|-----|
| 11 | **`_tick` silently swallows all exceptions without logging** — The streaming render loop at lines 297–321 has a bare `try/except` (lines 311–320) that catches `json.JSONDecodeError` and `TypeError` but produces no log output. Any rendering error is silently discarded, making debugging impossible for streaming display bugs. | `src/minimal_harness/client/built_in/app.py` (lines 311–320) | Add a `log.exception` or `print` call inside the except block, or replace the bare except with a guard that re-raises after logging. | Silent failures in a UI render loop hide production bugs and make it impossible to diagnose why streamed content is not appearing. |
| 12 | **Eager module-level imports inside `_banner`** — `get_bash_tools` and `get_patch_file_tools` are imported at lines 329–332 inside the `_banner` method, which is called on every app mount and after `/new`. These imports should be at the top of the file or cached after first load. | `src/minimal_harness/client/built_in/app.py` (lines 329–332) | Move the two import lines to module level, or cache the result of `get_bash_tools()` / `get_patch_file_tools()` in a module-level set the first time `_banner` runs. | Repeated import overhead on every `/new` command; more importantly, the import side-effects (module-level decorator registration in those modules) will fire multiple times. |
| 13 | **`AppContext.rebuild()` unconditionally creates a new `ConversationMemory` even when the system prompt is unchanged** — Lines 60–69 in `rebuild()` always replace `self.memory` if the system prompt in the config differs from the built-in default, but on many rebuild cycles (e.g., after tool selection changes that call `rebuild()`) the system prompt is unchanged, yet a new `ConversationMemory` is still allocated. | `src/minimal_harness/client/built_in/context.py` (lines 60–69) | Only replace `self.memory` if the prompt value has actually changed from the previously loaded prompt, not just different from the default. Store `self._last_prompt_path` and compare before allocating. | Unnecessary object churn; more critically, it breaks references to the old memory object that may be held by async tasks still running when `rebuild()` is called mid-stream. |
| 14 | **`StreamBuffer.render()` creates a new `rich.Console` per call** — The `render` method (buffer.py line 22–37) instantiates a fresh `Console` and `StringIO` on every invocation, which is called on every `_tick` (every 0.25 s) during streaming. | `src/minimal_harness/client/built_in/buffer.py` (lines 31–34) | Accept an optional `console: Console` parameter so callers can reuse a shared console instance, or maintain a console pool within `StreamBuffer` itself. | A new `Console` per 0.25 s tick is expensive (object allocation, ANSI detection, terminal capability probing). On slower terminals this could cause visible lag. |
| 15 | **`action_share` closure captures `rlog` by reference then mutates it** — The `done` callback (app.py line 531) captures `self._rlog` before `self.push_screen` schedules it. If `_banner` or another action fires before the screen is dismissed, `self._rlog` may have been replaced, causing the captured reference to be stale. | `src/minimal_harness/client/built_in/app.py` (lines 527–556) | Capture `self._rlog` inside the callback via a local binding (`rlog = self._rlog`) immediately before use rather than at the start of `done`, or pass it explicitly via a dataclass closure. | Stale reference risk; unlikely to fire in current single-threaded TUI but would break if `action_share` ever becomes non-blocking/async. |

## Resolved Issues

| # | Problem | File | Resolution |
|---|---------|------|------------|
| 20 | **Bash tool yielded exit code instead of actual output; final result not aggregated** — `bash_handler` yielded `f"[Exit code: {process.returncode}]"` as the final line and lost any buffered stdout/stderr after the loop break condition fired. Users saw "exit code: 0" instead of the command's actual output. | `src/minimal_harness/tool/built_in/bash.py` | Refactored to yield `dict` events: progress as `{"status": "progress", "message": ...}`, final event as `{"stdout": ..., "stderr": ...}`. All lines are collected in `stdout_lines`/`stderr_lines` lists and the final aggregated output is yielded after the reader tasks complete and the queue is drained. |
